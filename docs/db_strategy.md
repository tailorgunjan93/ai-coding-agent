# Database Strategy

## Overview
This document defines how the Database Node manages schema evolution, migrations, rollback, and ORM usage.  
It ensures data integrity, scalability, and maintainability across applications generated by the agent.

---

## Responsibilities
1. **Schema Management**
   - Define schema in `/db/schema.sql`.
   - Keep schema modular and normalized.
   - Document schema changes in `planning.md`.

2. **Migrations**
   - Store migration scripts in `/db/migrations/`.
   - Use migration tools (Flyway, Liquibase, Alembic, EF Migrations).
   - Ensure migrations are reversible.

3. **Rollback**
   - Every migration must include rollback scripts.
   - Rollback triggered automatically if tests fail.
   - Human approval required for rollback in production.

4. **ORM Usage**
   - ORM models stored in `/db/orm_models/`.
   - Supported ORMs:
     - Python → SQLAlchemy
     - .NET → Entity Framework
     - JS/TS → TypeORM/Prisma
   - Enforce repository pattern for DB access.

---

## Workflow Integration
- **Planning Stage**: DB requirements documented in `planning.md`.
- **Code Generation Stage**: DB Node generates schema + ORM models.
- **Testing Stage**: DB tests validate schema integrity and queries.
- **CI/CD Stage**: Pipelines run migrations automatically.
- **Rollback Stage**: Triggered if DB tests fail or deployment errors occur.

---

## Tools Used
- **Flyway / Liquibase** → Migration management.
- **SQLAlchemy / EF / TypeORM** → ORM handling.
- **Dockerized DBs** → Sandbox testing environments.

---

## Fail-Safe Rules
- Deployment blocked if:
  - Migrations are missing rollback scripts.
  - DB tests fail.
  - ORM models are inconsistent with schema.
- Overrides require explicit human approval documented in `planning.md`.

---

## Recommendations
- Always run migrations in sandbox before production.
- Document schema changes in `planning.md`.
- Treat DB handling as a **non-negotiable quality gate**.
